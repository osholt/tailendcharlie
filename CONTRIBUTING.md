# Contributing

Tail End Charlie is at an early, safety-sensitive stage. Small, testable changes are
preferred over broad feature additions.

## Development workflow

1. Open an issue for behaviour or architecture changes.
2. Create a focused branch.
3. Run `flutter analyze` and `flutter test` from `apps/mobile`.
4. Do not weaken offline, privacy, or uncertainty-state requirements to make a
   demo look connected.
5. Include field-test evidence for nearby radio or background-execution changes.

Never commit signing keys, provisioning profiles, credentials, precise field
ride traces, or real emergency-contact details.
